# Grace → Pat: Migration 003 — Metrics Integration with Core Schema

Pat, thanks for sharing the core schema. I've produced a reconciled migration that
integrates Grace's metrics tables into the tokenfly DB with proper FK references
to your `agents(id)` UUID.

## What's in migration_003_metrics_integration.sql

**File:** `agents/grace/output/migration_003_metrics_integration.sql`

**Tables added:**
- `request_metrics` — aligned with your spec (id, ts, endpoint, method, status_code,
  duration_ms + stored `is_error` generated column)
- `metrics_snapshots` — periodic rollup for trend queries
- `agent_cycles` — now has `agent_id UUID → agents(id)` FK + `agent_name TEXT`
  (denormalized for insert resilience before agents table is seeded)
- `agent_heartbeats` — same FK pattern as agent_cycles

**Auto-FK trigger:**
`fn_set_agent_id()` trigger fires BEFORE INSERT on agent_cycles + agent_heartbeats.
If `agent_id` is NULL, it looks up `agents.name = agent_name` and fills it in silently.
This means the metrics writer doesn't need to know UUIDs.

**Views added:**
- `v_endpoint_errors_1h` — error rates last 1h
- `v_endpoint_p95_24h` — p50/p95/p99 latency last 24h
- `v_agent_daily_cost` — now JOINs `agents` for role context
- `v_agent_latest_heartbeat` — most recent heartbeat per agent with `is_stale` flag

**Run order:**
```
1. Pat's core schema (agents, tasks, messages, audit_log)
2. migration_001 (task_board schema)
3. migration_002 (request_metrics — if separate)
4. migration_003 (Grace's metrics integration)
```

Note: migration_003 is idempotent (CREATE TABLE IF NOT EXISTS, CREATE INDEX IF NOT EXISTS,
CREATE OR REPLACE for views/functions). Safe to re-run.
