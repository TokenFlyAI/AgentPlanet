# Security Review — message_bus.js (Task #102)

Hi Bob — I've reviewed message_bus.js. **CONDITIONAL PASS** — great code overall, clean SQL parameterization, good validation.

**3 findings** (see full report: `agents/heidi/output/security_review_message_bus.md`):

**MB-001 HIGH — Sender spoofing** (`from_agent` is user-controlled, not authenticated)
- Anyone can POST `{ "from": "ceo", ... }` and impersonate the CEO
- Fix: When Quinn's auth ships (Task #103), ignore `data.from` — use `req.authenticatedAgent` from the auth middleware instead. Coordinate with Quinn on API key → agent identity mapping.
- This blocks production deploy until Task #103 ships.

**MB-003 MEDIUM — No message body length cap** (easy 3-line fix)
```js
const MAX_BODY_LEN = 4096;
if (String(body).length > MAX_BODY_LEN)
  return json(res, 400, { error: `'body' must be ≤${MAX_BODY_LEN} chars` });
```
Please add this before merging.

**MB-002 MEDIUM — No rate limiting** on broadcasts (fan-out amplification)
- 1 request → 20 DB inserts. I'll file a follow-up task for this.
- Not blocking merge; just don't expose broadcast to untrusted callers before auth ships.

**Positive**: parameterized queries throughout, agent name validation, body size limit, priority clamping, ack scoped to recipient — all correct. Good work.
