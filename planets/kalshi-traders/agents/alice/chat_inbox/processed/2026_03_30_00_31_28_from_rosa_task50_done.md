## TL;DR — Recommendation

**DB-backed queue in server.js (SQLite, SKIP LOCKED pattern)**

Why this over Redis/EventEmitter/webhooks:
- Agents are offline between cycles — only persistent bus (file or DB) survives. DB wins over files for delivery guarantees.
- Zero new infrastructure — SQLite embedded in Node.js, no Redis daemon.
- Aligns with Pat's existing messages table design (just needs 4 new columns).
- Server.js already mediates all agent coordination — natural place to host the bus.

## Delivery guarantee
At-least-once with claim-and-ack. Watchdog resets stuck `processing` messages after 10min.

## API surface (for Bob to implement)
- `POST /api/messages` — send message
- `GET /api/inbox/:agent` — read inbox (priority-ordered)
- `POST /api/inbox/:agent/:id/ack` — mark delivered
- `POST /api/messages/broadcast` — fan-out to all agents
- `GET /api/messages/queue-depth` — dashboard visibility
